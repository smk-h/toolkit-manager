# 项目级 MCP 配置管理 Checklist

> 每一项通过运行代码或观察行为来验证，聚焦系统行为。

## 实现完整性

- [ ] **[F1/AC1 入口]** 项目页激活时 Header 右侧出现「新增项目」按钮（验证：`pnpm tauri:dev` 启动，切换到项目页，观察 Header 右侧出现按钮且鼠标悬浮变橙色高亮）
- [ ] **[F1/AC1 新增]** 点击「新增项目」后，项目列表末尾追加一张空白卡片（验证：点击按钮，观察列表末尾新增空路径输入框的卡片）
- [ ] **[F2/AC2 打开按钮]** 点击卡片「打开」按钮弹出系统目录选择对话框（验证：点击按钮，观察系统目录选择器弹出；选中目录后输入框自动填充路径）
- [ ] **[F2/AC2 手动输入]** 用户可直接手动输入路径到输入框（验证：在输入框键入路径，观察值正常更新）
- [ ] **[F3/AC3 OK 态]** 面对已正确配置的项目（两文件齐备+字段正确+toolkit 正常），徽章显示绿色「OK」（验证：准备一个已配置好的项目目录，输入路径后等 400ms，观察徽章为绿色 OK）
- [ ] **[F3/AC3 未配置态]** 面对文件/字段缺失的项目（toolkit 正常），徽章显示橙色「未配置」（验证：准备一个无 .mcp.json 的空项目目录，输入路径，观察徽章为橙色「未配置」）
- [ ] **[F3/AC3 配置错误态]** 面对 toolkit 路径不存在或启动脚本缺失的情况，徽章显示红色「配置错误」（验证：在设置页把 toolkit 路径改成不存在的目录，回到项目页输入合法项目路径，观察徽章为红色「配置错误」）
- [ ] **[F4/AC4 一键配置]** 在「未配置」状态下点击配置按钮，缺失文件/字段被创建/补全，command 被修正（验证：对未配置项目点配置，用文件管理器/编辑器打开生成的 .mcp.json 与 .claude/settings.local.json，确认字段正确写入）
- [ ] **[F4/AC4 字段保留]** 一键配置不破坏两文件中的既有字段（验证：手动给项目目录下的 .mcp.json 添加一个自定义 server 或其他字段，点配置后重新打开文件，确认自定义字段仍在）
- [ ] **[F4/AC4 状态刷新]** 配置成功后徽章自动变为 OK（验证：点配置成功后，无需手动刷新，观察徽章自动变绿）
- [ ] **[F4/AC5 目录创建]** 项目目录不存在时点配置按钮，目录被创建后再写入配置（验证：输入一个不存在的项目路径，点配置，观察目录被创建且两文件生成）
- [ ] **[F4 配置按钮态]** 状态为 OK 时配置按钮灰色不可点击且无悬浮效果；非 OK 时可点击且有橙色悬浮（验证：对 OK 项目观察按钮灰色且鼠标悬浮无变化；对未配置项目观察按钮可点且悬浮变橙）
- [ ] **[F5/AC6 toolkit 引导]** toolkit 路径为空或不存在时，项目页显示引导卡片（验证：清空设置页 toolkit 路径或改为不存在路径，切到项目页，观察引导卡片显示）
- [ ] **[F5/AC6 跳转]** 点击引导卡片「去设置」按钮跳转到设置页（验证：点击按钮，观察左侧导航定位到设置页）
- [ ] **[F6/AC7 持久化]** 添加若干项目后重启应用，项目列表恢复（验证：添加 2-3 个项目路径，关闭应用重新打开，观察列表恢复如初）
- [ ] **[F7/AC8 重复警告]** 输入与已有项目相同的路径时出现重复警告（验证：两张卡片输入相同路径，观察第二张出现「路径与已有项目重复」红字）
- [ ] **[F8/AC9 不存在警告]** 输入磁盘不存在的路径时出现路径不存在警告，且该路径可被保留（验证：输入 `D:/nonexistent/xxx`，观察出现「路径不存在」红字；路径仍保留在输入框）
- [ ] **[F8/AC9 警告非阻断]** 路径不存在时仍可点配置按钮创建目录（验证：对不存在路径点配置，观察目录被创建、警告消失、徽章变 OK）
- [ ] **[F9/AC11 删除按钮]** 每张项目卡片有「删除」按钮且具备鼠标悬浮效果（验证：观察卡片右侧删除按钮，鼠标悬浮变橙）
- [ ] **[F9/AC11 删除记录]** 点击删除后卡片从列表消失，settings.json 中对应路径记录被移除（验证：点删除后观察列表；查看 settings.json 的 project-config 键确认记录已移除）
- [ ] **[F9/AC11 文件保护]** 删除项目卡片不删除/修改项目目录下的 .mcp.json 与 .claude/settings.local.json（验证：删除前记录两文件内容，删除后再次对比，内容完全一致）

## 集成

- [ ] **[透传链]** Header「新增项目」→ App state → ContentArea → ProjectsPage addItem 的完整链路打通（验证：点 Header 按钮，观察新卡片出现，证明四层透传无断裂）
- [ ] **[独立键]** 项目列表写入 settings.json 的独立键 `project-config`，与 `directory-config` 平级且互不干扰（验证：添加项目后在 `%APPDATA%\com.tauri-app.toolkit-manager\settings.json` 查看顶层有两个独立键，各自数据正确）
- [ ] **[toolkit 复用]** 项目页与设备页共用同一份 directory-config 的 toolkit 预置项路径（验证：设置页改 toolkit 路径后，项目页状态检测/配置行为随之变化）

## 编译与测试

- [ ] **[类型检查]** `pnpm build`（tsc + vite build）无错误（验证：运行命令，观察退出码 0、无类型错误）
- [ ] **[lint 风格]** 代码符合 ts-lang-spec 要求（验证：人工检查命名/风格/注释——文件头注释齐全、函数 JSDoc 完整、判别联合用法一致）
- [ ] **[文件编码]** 新建文件均为 UTF-8 无 BOM、LF 换行；修改的既有文件（store.ts/Header.tsx/ContentArea.tsx/App.tsx）保持原编码不变（验证：用编码检测工具或编辑器查看，无乱码、无 BOM 标记）

## 端到端场景

- [ ] **[E2E1 全流程]** 从零配置到 OK 的完整路径：设置页配置 toolkit 路径 → 项目页新增项目卡片 → 输入/选择一个空项目目录（路径不存在警告出现）→ 点配置按钮（目录创建+两文件写入）→ 徽章变 OK → 配置按钮变灰（验证：按顺序操作，每步观察对应预期结果）
- [ ] **[E2E2 持久化往返]** 添加多个项目 → 重启应用 → 列表恢复 → 其中一个项目的配置状态仍正确显示（验证：重启后观察列表与徽章状态）
- [ ] **[E2E3 字段保护]** 在已有 .mcp.json（含其他 server 与自定义字段）的项目上点配置 → 自定义字段保留、embedded-board 的 command 被修正 → 徽章变 OK（验证：点配置前后对比文件内容，自定义字段仍在）
- [ ] **[E2E4 降级健壮性]** 手动破坏一个项目的 .mcp.json（写入非法 JSON）→ 输入该路径 → 徽章降级为「未配置」而非崩溃 → 点配置按钮后文件被修正为合法 JSON（验证：破坏文件后观察页面不崩溃、徽章正确、配置后文件恢复合法）
