# 设置页目录配置 Spec

## 一、背景

### 1.1 当前状态

toolkit-manager 经 ch01（界面骨架）与 ch02（设备卡片）两轮迭代后，已具备三栏布局（Header + SideNav + ContentArea）、四个 Tab 路由（设备/项目/设置/关于）、shadcn 风格基础组件（tooltip / button / card）。但四个页面中只有"关于"和"设备"有真实内容，"项目"和"设置"仍是 `PlaceholderPage` 占位。

"设置"页是用户配置应用行为的核心入口。ch01 `user.md` 第 247 行已规划"第四章：应用设置（目录配置、主题切换、导入导出）"，其中**目录配置**是应用设置的基础能力——许多后续功能（如扫描项目、读写日志、缓存、备份）都依赖用户预先指定的目录路径。

### 1.2 要解决的问题

让用户在设置页维护一个**目录路径列表**，每项代表一个用户关心的本地目录。用户可以：

1. **手动输入路径**——直接键入绝对路径保存
2. **打开目录选择器**——点击按钮调用系统文件夹选择对话框，选中后自动填充路径
3. **动态增删**——不限数量地新增目录项、删除已有目录项

同时需要对路径做**存在性校验**，引导用户填入有效路径，但不强制阻断保存（允许"先配置后建目录"）。

### 1.3 技术约束

当前项目 Tauri 环境中：

- **dialog 插件未安装**（package.json / Cargo.toml / capabilities 三处均缺）——"打开目录"按钮需新增依赖
- **fs 插件未安装**——目录存在性校验需新增依赖
- **store 插件未安装**——持久化需新增依赖
- **表单 UI 组件缺失**——`src/components/ui/` 仅有 button/card/tooltip，没有 input/label

因此本章除业务实现外，还包含 **Tauri 插件接入（dialog / fs / store 三件套）** 与 **基础 UI 组件补齐** 两项基础设施工作。

## 二、目标

- G1：让"设置"页从占位页升级为目录配置页，用户可维护一个目录路径列表
- G2：支持两种路径录入方式——文本输入 + 系统文件夹选择器
- G3：支持动态增删目录项（不限数量）
- G4：对路径做存在性实时校验，无效路径给出提示但不阻断保存
- G5：配置即时持久化到 Tauri Store 插件（应用 cache 目录的 JSON 文件），刷新与重启后恢复
- G6：接入 Tauri dialog / fs / store 三件套插件，补齐 input/label 基础 UI 组件，为后续章节（主题切换、导入导出）打基础

## 三、功能需求

### F1 设置页改造为目录配置页

**行为：** 用户点击侧边栏"设置"Tab，内容区不再显示占位页，而是渲染目录配置界面。

**界面结构（自上而下）：**

| 区域 | 内容 | 说明 |
|------|------|------|
| 页面标题 | "应用设置" + 副标题"目录配置" | 与 ch02 设备页标题区风格一致 |
| 说明文案 | 一句话说明用途，如"管理应用关注的本地目录" | 灰色小字 |
| 目录卡片 | 一个卡片容器，内含目录项列表 + 底部"添加目录"按钮 | 用 ch02 沉淀的 `Card` 组件族 |
| 空状态 | 列表为空时显示"暂无目录，点击下方添加"提示 | 居中、虚线边框 |

**约束：**

- 保留 `SettingsPage` 的导出签名（被 `ContentArea` 引用，签名变化会破坏路由）
- 整体布局沿用 ch02 设备页的标题区 + 内容区分离风格

### F2 目录项展示与录入

**行为：** 每个目录项是一行横向布局，包含：

| 元素 | 内容 | 交互 |
|------|------|------|
| 自动名称 | 从路径末尾目录名提取（如 `D:/foo/bar` → `bar`）；路径为空时显示"未命名" | 只读展示 |
| 路径输入框 | 文本输入，显示当前路径 | 可编辑，onChange 即时保存 |
| 打开目录按钮 | 图标按钮（FolderOpen 图标） | 点击调用系统文件夹选择器 |
| 删除按钮 | 图标按钮（Trash2 图标） | 点击删除该项 |
| 校验提示 | 路径无效时输入框下方红字"目录不存在" | 实时校验触发，路径有效时隐藏 |

**约束：**

- 自动名称**不可编辑**，完全由路径派生（用户改路径，名称跟随变化）
- 名称提取规则：取路径按 `/` 或 `\` 分割后的最后一段非空部分；路径为空或仅分隔符时显示"未命名"
- 路径输入框宽度自适应，撑满可用空间
- 三个按钮（打开/删除）紧凑排列在输入框右侧

### F3 动态增删

**行为：**

- **新增：** 卡片底部有"添加目录"按钮（Plus 图标 + 文案），点击后在列表末尾追加一个空路径项，并自动聚焦其输入框
- **删除：** 每个目录项右侧的删除按钮，点击立即删除该项（无需二次确认，因为操作可逆——重新添加即可）

**约束：**

- 新增项的初始路径为空字符串，名称显示"未命名"，校验提示"目录不存在"（空路径视为无效）
- 不限数量，但列表为空时只显示空状态提示与"添加目录"按钮
- 删除后剩余项的顺序保持不变（不重排）

### F4 打开目录选择器

**行为：** 点击目录项的"打开目录"按钮：

1. 调用系统原生文件夹选择对话框（Tauri dialog 插件，`directory: true` 模式）
2. 用户选择一个目录并确认 → 该项的路径输入框自动填充所选路径 → 触发 onChange 即时保存 → 触发实时校验
3. 用户取消（关闭对话框）→ 无任何变化

**约束：**

- 对话框标题为"选择目录"
- 仅允许选择目录（`directory: true`），不允许选文件
- 对话框未确认前不阻塞 UI（Tauri 的 `open()` 是异步的，await 期间用户可继续操作其他项）

### F5 路径存在性实时校验

**行为：** 路径输入框内容变化时（无论是手动输入还是选择器填充），实时校验目录是否存在：

- **校验方式：** 调用 Tauri fs 插件检查路径是否存在且为目录
- **防抖：** 输入连续变化时 debounce 400ms，避免逐字符触发 fs 调用
- **反馈：** 路径无效（不存在或非目录）时，输入框下方显示红字"目录不存在"；路径有效时隐藏提示
- **空路径：** 视为无效，但不显示红字（空路径是"未填写"而非"填错"，UI 仅在用户填写过又清空时才提示）

**约束：**

- **校验失败不阻断保存**——即使路径无效，仍写入 localStorage（支持"先配置后建目录"场景）
- 校验是**纯视觉提示**，不改变保存行为
- 路径为空字符串时：不调用 fs、不显示"目录不存在"（避免新增空项时一片红字）

### F6 即时持久化（Tauri Store 插件）

**行为：** 任何导致目录列表变化的操作（新增/删除/路径修改）都立即持久化到 Tauri Store 插件：

- 新增项 → 追加到列表并写入 Store
- 删除项 → 从列表移除并写入 Store
- 路径变化 → 更新对应项并写入 Store

**技术方案：** 使用 Tauri 官方 Store 插件（`@tauri-apps/plugin-store`），配置持久化到应用 cache 目录下的 JSON 文件。

**约束：**

- Store 文件名：`settings.json`（应用级设置统一存这一份，后续章节的主题切换、导入导出配置也复用此文件，按 key 区分）
- 存储键：`directory-config`，值为 JSON 字符串数组（每项为路径字符串，名称不入库，由路径派生）
- 写入采用 `autoSave: false` + 手动 `store.save()`，确保每次变更显式落盘（避免内存态与磁盘态不一致）
- 写入失败时静默降级（不阻断 UI），与 localStorage 模式精神一致

### F6.1 异步加载与初始化态

**行为：** 由于 Store 插件所有操作均为异步（Promise），与同步的 localStorage 不同，应用启动时需异步加载配置：

- **加载时机：** 设置页首次渲染时（或应用启动时预加载）通过 `useEffect` 异步读取 Store
- **加载态：** 加载完成前，目录列表显示为空 + 骨架/加载提示（避免闪现空状态再跳变）；加载完成后填充真实数据
- **加载失败：** Store 读取异常时降级为空列表（与首次使用无异），并在控制台记录错误

**约束：**

- 加载态需对用户可见（不能"假装已加载"显示空列表，造成"我的配置丢了"的误解）
- 加载完成后用户的后续操作（增删改）走"读内存 → 写内存 → 写 Store"流程，不阻塞 UI

### F7 基础设施补齐

**行为：** 为支撑上述功能，补齐三项基础设施：

- **Tauri 插件接入**：安装并启用 dialog、fs、store 三个插件（前端 npm 依赖 + Rust Cargo 依赖 + capabilities 权限 + Rust 侧插件注册）
- **UI 组件补齐**：新增 `input.tsx`、`label.tsx` 两个 shadcn 风格基础组件，供目录配置表单使用

**约束：**

- 三个插件按 Tauri v2 标准接入方式（四处同步修改：npm / Cargo / capabilities / lib.rs）
- 修改 `package.json` / `Cargo.toml` / `tauri.conf.json` / `capabilities/default.json` / `src-tauri/src/lib.rs` 时**保持原文件编码与换行符不变**（这些文件多为 CRLF，须用工具操作而非手编）
- input/label 组件遵循 shadcn 官方实现风格，与现有 button/card 风格一致

### N2 依赖与主题

### N5 文件编码

- 新建文件：UTF-8 无 BOM、LF 换行、文件末尾保留一个空行
- 修改已有文件：保持原编码与换行符不变（特别注意 `package.json` / `Cargo.toml` / `tauri.conf.json` / `capabilities/default.json` / `lib.rs` 的 CRLF）

### N6 可访问性

- 路径输入框有关联的 `<label>`（通过 `htmlFor` / `id` 关联）
- 按钮有 `aria-label` 或可见文案
- 校验提示用 `aria-invalid` 与 `aria-describedby` 关联到输入框，供屏幕阅读器播报
- 错误提示颜色不仅靠红色，辅以图标或文字语义

### N7 错误降级

- Tauri Store 读写异常静默降级（读失败降级为空列表、写失败不阻断 UI）
- Tauri dialog / fs 调用失败（如权限被拒、路径含特殊字符）时，UI 不崩溃，错误提示降级为"无法校验"或保持上次状态
- 在非 Tauri 环境（如纯浏览器 dev）调用 dialog / fs / store 应优雅降级——给出提示或禁用按钮，不抛未捕获异常

## 五、不做的事

### 5.1 明确排除的功能

- **不做目录名称自定义**——名称完全由路径派生（spec 5.2 排除项，对应 F2 约束）
- **不做删除二次确认**——删除直接生效（F3 约束）
- **不做拖拽排序**——目录项顺序按新增顺序，不重排（F3 约束）
- **不做目录内容预览/浏览**——只存路径，不读取目录内文件
- **不做批量导入/导出**——配置仅本地持久化，不提供 JSON 导入导出（留给后续"导入导出"章节）
- **不做路径补全/建议**——输入时不弹候选目录列表

### 5.2 明确排除的校验

- **不校验路径格式合法性**（如非法字符）——仅校验"目录是否存在"，格式问题由 fs 接口自然反馈
- **不校验目录权限**（可读/可写）——仅校验存在性
- **不校验目录是否为空**——空目录也是合法配置

### 5.3 明确排除的扩展

- **不做主题切换、导入导出等其他设置项**——本章只做目录配置，其他设置留给后续章节
- **不重构设置页为多 Tab**——当前设置页只有目录配置一项内容，无需 Tab 切换
- **不做配置项的"启用/禁用"开关**——目录项要么存在要么删除，无中间态
